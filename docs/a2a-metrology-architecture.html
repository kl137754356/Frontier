② 一次测量的 Task 流程（含安全阀门）

1
接收意图Orchestrator

用户「测一下这个零件的孔径」→ 解析为测量计划草案（程序名 / 特征 / 公差）。

🛡
安全阀门 A · 运动前置审查→ Safety Agent

检查碰撞域、探针速度、零件是否夹紧、是否超行程。通过则签发带签名的放行 token；否则中止并说明原因。

👤
安全阀门 B · 人在回路（高风险才触发）A2UI present_choices

首件 / 改程序 / 超行程等高风险动作，弹确认卡片给操作员，复用现有 A2UI。低风险自动跳过。

2
执行测量→ Executor Agent

校验放行 token 有效 → 通过 MCP 调 PC-DMIS COM 执行，流式回传进度。无 token 一律拒绝。

3
独立复核→ Analyst Agent

对结果做公差判定、重复性 / GR&R、与历史趋势比对，标记异常点，给出合格 / 复测 / 不合格结论。

↺
判异分支Orchestrator 决策

不合格或可疑 → 触发复测（回到步骤 1，复用同一安全链）或上报人工。合格 → 进入下一步。

4
汇总与留痕Orchestrator + Telemetry

结果回传前端；task id / 输入 / 放行记录 / 复核结论写入 telemetry（local-store），可追溯。

③ 各 Agent 职责与协议

🧭 Frontier Orchestrator A2A Client

  • 面向用户，理解意图与编排
  • 按 task 链调用下游 Agent
  • 判异分支决策（复测 / 上报）
  • 不直接接触硬件

🛡 Safety / Guard A2A Server

  • 运动前置审查规则引擎
  • 签发 / 校验放行 token
  • 判断是否需人在回路
  • 可独立部署、独立升级

⚙ Measurement Executor A2A MCP

  • 唯一持硬件访问权的 Agent
  • 无有效放行 token 拒绝执行
  • 通过 MCP 调 PC-DMIS COM
  • 流式回传执行进度

🔬 Metrology Analyst A2A Server

  • 公差 / 重复性 / 趋势分析
  • 对接 Orca 8780 辅助数据
  • 给出独立第二意见
  • 只读，不触发物理动作

④ 安全设计要点

维度做法
Agent 间鉴权A2A 之间用 bearer token / mTLS，Agent Card 声明 securitySchemes，本机也不裸跑。
物理动作闸门Executor 收到运动类 task 必须携带 Safety 签发的放行 token，校验失败即拒。
人在回路高风险动作通过 A2UI present_choices 弹操作员确认，复用现有组件。
审计留痕每个 task 的 id / 输入 / 放行 / 复核结论写入 telemetry local-store.ts。
输入隔离其他 Agent 的返回视为不可信数据，不得直接触发新的物理动作。
权限最小化仅 Executor 持硬件权；Analyst / Safety 无任何驱动探针的能力。

⑤ 增量落地路径

阶段交付物说明
阶段一PC-DMIS MCP Server把现有 probe/state 脚本封成工具，复用 mcp-server 的 Content-Length 写法。独立可用。
阶段二Executor A2A 外壳用 @a2a-js/sdk 起 server，发布 Agent Card，把 MCP 工具映射为 A2A skill。
阶段三Safety + Analyst Agent加入安全闸门与独立复核，串成完整 task 链与判异回路。
阶段四审计 + 人在回路接 telemetry 留痕，高风险接 A2UI 确认，完成闭环。
